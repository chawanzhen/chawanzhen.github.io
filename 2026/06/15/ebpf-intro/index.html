eBPF 入门：在内核里跑沙盒程序是什么体验

什么是 eBPF

eBPF（extended Berkeley Packet Filter）是 Linux 内核中的一项技术，允许你在不修改内核源码、不加载内核模块的前提下，安全地运行一段沙盒化程序。

换句话说：你写一段 C 代码，编译成 eBPF 字节码，通过 bpf() 系统调用加载到内核，内核的验证器检查它是否安全，然后把它挂到某个内核钩子上——每当那个钩子被触发，你的代码就执行。

为什么 eBPF 这么火

传统的内核开发路线：改内核 → 编译 → 重启 → 测试。一个周期可能半小时起步。

eBPF 的路线：写 eBPF 程序 → bpftool 加载 → 立刻运行。秒级迭代。

这让很多以前只能靠内核模块做的事情，现在可以用 eBPF 安全、快速地完成：

场景 传统方案 eBPF 方案
网络包过滤 iptables/netfilter XDP/TC
系统调用监控 auditd tracepoint/kprobe
性能分析 perf eBPF tracing
安全防护 LSM 模块 LSM + eBPF
容器网络 iptables 规则 Cilium (eBPF)

核心概念

Hook 点

eBPF 程序需要挂载到内核的某个钩子上才能被触发。常见的钩子类型：

  • kprobe / kretprobe — 动态插桩，挂到任意内核函数
  • tracepoint — 内核预定义的静态追踪点（更稳定）
  • XDP — 网卡驱动层，比任何内核网络栈都早
  • TC — 流量控制层，在数据包进出网卡时
  • LSM — Linux 安全模块，做访问控制

验证器

内核在执行 eBPF 程序之前，会先跑验证器。验证器做这些检查：

  1. 无死循环 — 不能有无法退出的循环（内核 5.3+ 支持有界循环）
  2. 无非法内存访问 — 所有指针都要检查是否在合法范围
  3. 调用指令合法 — 只能调用白名单里的 helper 函数
  4. 栈深度限制 — 最多 512 字节的栈空间
  5. 指令数限制 — 最初 4096 条，5.1+ 放宽到 100 万条

验证器是 eBPF 安全模型的基石。没有验证器，加载到内核的代码可以为所欲为——那 eBPF 就成了后门。

Map

eBPF 程序之间、用户态和内核态之间靠 BPF Map 交换数据。Map 是内核中的键值存储，类型很丰富：

  • BPF_MAP_TYPE_ARRAY — 固定大小数组
  • BPF_MAP_TYPE_HASH — 哈希表
  • BPF_MAP_TYPE_RINGBUF — 环形缓冲区（发送事件到用户态）
  • BPF_MAP_TYPE_PERCPU_ARRAY — 每个 CPU 私有的数组

学习路线

如果有 C 语言和 Linux 基础，推荐这条路线：

  1. 看懂验证器 — 理解检查逻辑，知道什么代码能通过、什么会被拒
  2. 写几个简单 tracepoint 程序 — 挂到 sys_enter_openat、sys_enter_execve 之类的地方
  3. 学会用 bpftool / libbpf — 告别手动 bpf() 系统调用
  4. 尝试 TC / XDP — 搞点网络相关的
  5. 研究 Cilium — 看看工业级 eBPF 项目怎么组织代码

总结

eBPF 把内核变成了一个可编程的平台。它不像内核模块那样危险，但比用户态工具强大得多。

如果你对「在内核里跑沙盒程序」这件事感到好奇，建议从 bpf_printk + tracepoint 开始，几分钟就能看到效果。
